# DoctorLink 의료 예약 플랫폼 졸업논문

## I. 서론

### 1. 개발 배경

현대 사회에서 의료 서비스는 국민 건강의 핵심 요소이지만, 전통적인 병원 예약 시스템은 여러 불편함을 내포하고 있다. 환자들은 병원 예약을 위해 전화 통화를 해야 하며, 진료 가능 시간을 실시간으로 확인하기 어렵고, 예약 변경이나 취소 과정이 복잡하다. 또한 의료 기록이 분산되어 있어 환자가 자신의 진료 이력을 체계적으로 관리하기 어려운 실정이다.

특히 코로나19 팬데믹 이후 비대면 서비스의 중요성이 부각되면서, 의료 분야에서도 디지털 전환의 필요성이 대두되었다. 환자들은 24시간 언제든지 병원을 검색하고 예약할 수 있는 플랫폼을 필요로 하며, 의료 기관은 예약 관리의 효율성을 높이고 노쇼(No-show) 문제를 해결해야 하는 과제를 안고 있다.

이러한 배경에서 본 프로젝트는 웹과 모바일을 아우르는 통합 의료 예약 플랫폼 "DoctorLink"를 개발하여, 환자와 의료진 간의 연결을 원활하게 하고 의료 서비스 접근성을 향상시키는 것을 목표로 한다.

### 2. 개발 목적

본 프로젝트의 구체적인 개발 목적은 다음과 같다:

**2.1 환자 편의성 증대**
- 24시간 온라인 병원 예약 서비스 제공
- GPS 기반 주변 병원 검색 및 위치 안내
- 진료 기록 통합 관리 및 언제든 조회 가능
- 예약 알림을 통한 노쇼 방지

**2.2 의료 기관 운영 효율화**
- 자동화된 예약 시스템으로 인건비 절감
- 실시간 예약 현황 파악 및 관리
- 환자 데이터베이스 구축으로 맞춤형 서비스 제공
- 통계 기반 의료 자원 최적화

**2.3 의료 정보 비대칭 해소**
- 병원 및 의사 정보 투명한 공개
- 환자 리뷰 및 평점 시스템
- 진료과별 전문의 정보 제공
- 대기 시간 및 진료비 사전 안내

**2.4 기술적 목표**
- React/React Native 기반 크로스플랫폼 개발
- RESTful API 아키텍처 구현
- 보안성 높은 JWT 인증 시스템
- 확장 가능한 데이터베이스 설계

### 3. 한계점 및 진행방향

**3.1 현재 한계점**

**기술적 한계**
- 실제 의료 기관과의 연동 미구현 (프로토타입 단계)
- 결제 시스템 미연동 (PG사 연동 필요)
- 실시간 푸시 알림 미완성 (FCM 연동 예정)
- 대규모 트래픽 처리 미검증

**법적/제도적 한계**
- 의료법상 원격 진료 제한
- 개인정보보호법 완전 준수 필요 (PIPA)
- 의료 데이터 보안 규제 (HIPAA 수준 필요)
- 의료 광고 규제 준수

**사업적 한계**
- 의료 기관 네트워크 구축 필요
- 수익 모델 검증 필요 (광고, 수수료 등)
- 초기 사용자 확보 전략 부족

**3.2 향후 진행방향**

**단기 계획 (6개월)**
1. Firebase Cloud Messaging 연동으로 실시간 푸시 알림 구현
2. 카카오페이, 토스페이먼츠 등 결제 시스템 연동
3. 실제 의료 기관 1~2곳과 파일럿 테스트 진행
4. 보안 강화 (SSL 인증서, 데이터 암호화)

**중기 계획 (1년)**
1. AWS/Azure 클라우드 인프라로 마이그레이션
2. CI/CD 파이프라인 구축 (GitHub Actions, Jenkins)
3. 로드 밸런서 도입으로 부하 분산
4. Redis 캐싱으로 성능 최적화
5. 사용자 피드백 기반 UI/UX 개선

**장기 계획 (2~3년)**
1. AI 기반 증상 분석 및 병원/의사 추천 시스템
2. 블록체인 기술 활용 의료 기록 무결성 보장
3. 텔레메디신(원격 진료) 기능 추가 (법적 허용 시)
4. 다국어 지원 및 해외 의료 관광 연계
5. 웨어러블 기기 연동 (Apple Watch, Galaxy Watch)

### 4. 전공 이론

본 프로젝트는 컴퓨터공학 전공 과정에서 학습한 다양한 이론을 실무에 적용하였다.

**4.1 소프트웨어 공학 (Software Engineering)**
- **생명주기 모델**: 반복적 개발 방법론(Iterative Development) 적용
- **요구사항 분석**: 사용자 스토리 기반 기능 정의
- **설계 패턴**: MVC, Singleton, Observer 패턴 활용
- **테스팅**: 단위 테스트, 통합 테스트 전략 수립

**4.2 데이터베이스 (Database)**
- **관계형 데이터베이스**: MS SQL Server 활용
- **정규화**: 제3정규형(3NF)까지 적용하여 데이터 무결성 확보
- **인덱싱**: 검색 성능 향상을 위한 인덱스 설계
- **트랜잭션**: ACID 원칙 준수, 동시성 제어
- **저장 프로시저**: 복잡한 비즈니스 로직의 DB 레벨 처리

**4.3 네트워크 및 웹 기술**
- **HTTP/HTTPS 프로토콜**: RESTful API 통신
- **클라이언트-서버 아키텍처**: 3-tier 구조
- **세션 관리**: JWT(JSON Web Token) 기반 무상태 인증
- **CORS**: 크로스 오리진 리소스 공유 설정

**4.4 알고리즘 및 자료구조**
- **검색 알고리즘**: 병원 검색 시 문자열 매칭
- **정렬 알고리즘**: 거리순, 평점순, 이름순 정렬
- **해시 테이블**: 빠른 데이터 접근을 위한 Map 구조
- **트리 구조**: 계층적 메뉴 및 카테고리 관리

**4.5 정보 보안**
- **암호화**: bcrypt를 통한 비밀번호 해싱
- **인증/인가**: JWT 토큰 기반 접근 제어
- **입력 검증**: SQL Injection, XSS 공격 방어
- **개인정보 보호**: 의료 데이터 암호화 저장

**4.6 모바일 애플리케이션**
- **Android/iOS 플랫폼**: React Native로 크로스플랫폼 개발
- **GPS 위치 서비스**: Geolocation API 활용
- **푸시 알림**: FCM(Firebase Cloud Messaging) 계획
- **오프라인 저장**: AsyncStorage, SQLite 활용

**4.7 인간-컴퓨터 상호작용 (HCI)**
- **사용자 중심 설계**: 페르소나 기반 UI/UX 디자인
- **접근성**: WCAG 2.1 가이드라인 준수
- **반응형 디자인**: 다양한 화면 크기 대응
- **피드백 설계**: 사용자 행동에 대한 즉각적 응답

### 6. 기능

본 프로젝트는 웹과 모바일 플랫폼에서 다음과 같은 기능을 제공한다.

**6.1 인증 및 사용자 관리**
- 이메일 기반 회원가입 및 로그인
- JWT 토큰 기반 인증 시스템
- 카카오 소셜 로그인 (계획)
- 비밀번호 찾기 및 재설정

  - 카카오맵 연동 위치 표시

  - 처방전 정보

**6.6 마이페이지**

### 8. 팀 조직도

본 프로젝트는 1인 개발로 진행되었으며, 역할은 다음과 같이 분담하였다.

```
프로젝트 매니저 (PM)
└── 김태정 (컴퓨터공학과 4학년)
    ├── 기획자
    │   ├── 요구사항 분석
    │   └── 프로젝트 일정 관리
    │
    ├── 디자이너
    │   ├── UI/UX 설계
    │   └── 프로토타입 제작
    │
    ├── 백엔드 개발자
    │   ├── API 서버 개발
    │   ├── 데이터베이스 설계
    │   └── 인증 시스템 구현
    │
    ├── 웹 프론트엔드 개발자
    │   ├── React 웹 앱 개발
    │   └── 관리자 페이지 개발
    │
    ├── 모바일 개발자
    │   ├── React Native 앱 개발
    │   └── Android/iOS 빌드
    │
    └── 테스터/배포 담당자
        ├── 기능 테스트
        ├── 서버 배포
        └── 문서화
```

**개인 역량 강화를 위한 학습 분야:**
- 풀스택 개발 역량 (Front-end + Back-end)
- 크로스플랫폼 모바일 개발
- RESTful API 설계 및 구현
- 데이터베이스 설계 및 최적화
- UI/UX 디자인 기초
- 프로젝트 관리 및 일정 수립
